# Карты несовершеннолетних — веб-приложение

[![License: GPL v3](https://img.shields.io/badge/License-GPLv3-blue.svg)](LICENSE)
[![Python](https://img.shields.io/badge/Python-3.x-3776AB.svg?logo=python&logoColor=white)](https://www.python.org/)
[![Flask](https://img.shields.io/badge/Flask-2.3%2B-000000.svg?logo=flask&logoColor=white)](https://flask.palletsprojects.com/)

Веб-версия двух учётных форм:
- **N 030/у-Д/с** — «Карта диспансеризации несовершеннолетнего»;
- **N 030-ПО/у** — «Карта профилактического медицинского осмотра несовершеннолетнего».

Позволяет заполнять карты в браузере, сохранять их в базе SQLite (`base.db`),
редактировать и распечатывать (или сохранять в PDF средствами браузера).

## Возможности
- Две формы с переключением через **вкладки** на главной странице. Каждая
  вкладка показывает только свои карты; кнопка «Новая карта» создаёт форму
  текущей вкладки. Рядом с вкладками — пункт **«⚙ Настройки»**.
- Полное воспроизведение разделов официальных форм + подписи.
- **Динамические диагнозы** (минимум 1, максимум 10, кнопка «+ Добавить диагноз»),
  с автоматическим пересчётом номеров пунктов и зависимых разделов:
  - форма 030/у-Д/с — разделы 16 и 17;
  - форма 030-ПО/у — разделы 15 и 16.
- Большие авто-растущие текстовые поля в разделах с рекомендациями/перечнями.
- **Серверное автосохранение** через 2 секунды после изменения: новая карта
  становится черновиком, а в панели отображается время последнего сохранения.
  Медицинские данные не записываются в локальное хранилище браузера.
- **Совместная работа 3–5 пользователей**: SQLite работает в режиме WAL с
  ожиданием кратковременной блокировки; разные карты можно заполнять одновременно.
- **Защита одной карты от одновременной перезаписи**: каждая запись имеет номер
  ревизии, и устаревшая вкладка получает предупреждение вместо потери чужих правок.
- Авто-маска дат (дд.мм.гггг) и СНИЛС (ХХХ-ХХХ-ХХХ ХХ) с проверкой корректности
  **и в браузере, и на сервере** (некорректные данные в базу не попадают).
- **Настройки**: реквизиты медицинской организации вводятся один раз и
  автоматически подставляются в шапку каждой новой карты (обеих форм).
- **CRUD карт**: создание, редактирование, удаление; хранение всех карт в
  `base.db` (с указанием типа формы).
- **Раздельная порядковая нумерация** карт в каждом списке (своя для 030/у-Д/с
  и для 030-ПО/у), без «дыр» после удаления.
- **Корзина**: удаление мягкое — карта перемещается в корзину, откуда её можно
  восстановить или удалить навсегда.
- Печать заполненной карты под формат A4 (кнопка «Печать» → в диалоге можно
  выбрать «Сохранить как PDF»):
  - вёрстка под печать без лишних пустот и «рваных» рамок при переносе
    диагнозов между страницами;
  - печать всегда открывается в **одной и той же вкладке** (не плодит новые).

## Авторизация

Все страницы защищены входом по логину и паролю (сессия хранится в cookie).

- При первом запуске создаётся учётная запись по умолчанию:
  **логин `admin`, пароль `admin`**.
- Секретный ключ сессий генерируется **заново при каждом запуске** сервера,
  поэтому после перезапуска сервера/процесса требуется **повторный вход**.
- Сессия завершается автоматически после **2 часов бездействия** (срок продлевается
  при активности). Настраивается константой `SESSION_TIMEOUT` в `app.py`.
- **Смените пароль по умолчанию** прямо в интерфейсе: нажмите на **имя
  пользователя** в правом верхнем углу (на странице списка или в открытой
  форме) → откроется страница смены пароля.
- Добавить нового пользователя можно командой (из папки WebApp):

  ```powershell
  python -c "import sqlite3; from werkzeug.security import generate_password_hash as h; db=sqlite3.connect('base.db'); db.execute('INSERT INTO users (username,password_hash) VALUES (?,?)', ('user2',h('пароль2'))); db.commit(); print('пользователь добавлен')"
  ```

## Настройки

Пункт **«⚙ Настройки»** в меню (рядом с вкладками форм) позволяет задать
**значения по умолчанию** для часто повторяющихся полей. Что заполнено —
автоматически подставляется в **новые** карты (поля остаются редактируемыми);
пустые поля не подставляются; на уже сохранённые карты не влияет.

Предзаполняемые поля:
- **Общие (обе формы):** реквизиты медорганизации (ОГРН и т.д.);
- **030/у-Д/с:** наименование/адрес учреждения (п. 1, 1.2), МО первичной помощи
  (п. 9, 10), МО, проводившей диспансеризацию (п. 12);
- **030-ПО/у:** МО первичной помощи (п. 6, 7), образовательная организация
  (п. 8, 9), МО, проводившая осмотр (п. 11).

**Выпадающие списки с возможностью ручного ввода** (через HTML `datalist`) —
наборы значений задаются в «Настройках» (по одному в строке):
- «Страховая медицинская организация» (обе формы);
- Ф.И.О. врача, ответственного за проведение (подпись);
- Ф.И.О. руководителя медицинской организации (подпись).

Все настройки хранятся в таблице `settings` (ключ → значение).

## Установка и запуск (Windows, PowerShell)

```powershell
git clone https://github.com/diablotaurus/med-forms-030.git
cd med-forms-030
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
python app.py
```

Затем открыть в браузере: <http://127.0.0.1:5000>

> ⚠️ `python app.py` — это сервер **разработки**. Для постоянной работы на
> сервере используйте production-запуск (waitress) — см. ниже.

## Развёртывание на сервере (Windows Server)

Для боевого использования приложение запускается через WSGI-сервер
**waitress** (точка входа `serve.py`), при необходимости — как служба Windows.
Готовые скрипты установки, запуска и **обновления с GitHub** лежат в папке
[`scripts/`](scripts/) — подробная инструкция в [`scripts/README.md`](scripts/README.md).

Кратко:

```powershell
git clone https://github.com/diablotaurus/med-forms-030.git
cd med-forms-030
powershell -ExecutionPolicy Bypass -File scripts\setup.ps1   # установка
powershell -ExecutionPolicy Bypass -File scripts\start.ps1   # запуск
```

Обновление развёрнутого приложения до последней версии из репозитория:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\update.ps1
```

Для ежедневной автоматической проверки в 03:00:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\install-update-task.ps1 -Time "03:00"
```

База `base.db` создаётся автоматически при первом запуске рядом с `app.py`.

## Развёртывание на Linux

Для Ubuntu Server 24.04 LTS и Debian 12 подготовлен отдельный набор сценариев
в [`scripts/linux/`](scripts/linux/). Рекомендуемая конфигурация одного
независимого сервера: **2 vCPU, 2 ГБ RAM и 25 ГБ SSD**.

Краткая установка:

```bash
sudo apt update
sudo apt install -y python3 python3-venv git
sudo git clone https://github.com/diablotaurus/med-forms-030.git /opt/med-forms-030
sudo chown -R "$USER":"$USER" /opt/med-forms-030
cd /opt/med-forms-030
scripts/linux/setup.sh
sudo scripts/linux/install-service.sh
```

Приложение запускается как служба `systemd` и автоматически стартует после
перезагрузки. Ручное обновление с GitHub:

```bash
cd /opt/med-forms-030
scripts/linux/update.sh
```

Полная инструкция: [`scripts/linux/README.md`](scripts/linux/README.md).
Каждый сервер использует собственную локальную `base.db`; общий файл SQLite
между серверами не применяется.

Автоматическая ночная проверка на Linux включается командой:

```bash
sudo scripts/linux/install-update-timer.sh 03:00
```

## База данных (`base.db`)
- `forms` — карты: `id`, `patient_fio`, `birth_date`, `form_type`,
  `deleted` (0 — активна, 1 — в корзине), `created_at`, `updated_at`,
  `is_draft`, `revision`, `updated_by`, `data` (все поля формы в JSON).
  Номер в списке — порядковый (per type), не равен `id`;
- `users` — пользователи: `username`, `password_hash`;
- `settings` — настройки вида ключ→значение (например, `org_info`).

При запуске выполняется лёгкая миграция: недостающие столбцы/таблицы
добавляются автоматически, поэтому существующая база обновляется без потерь.
Для устойчивой работы нескольких пользователей включается SQLite WAL и
`busy_timeout=10000` мс. Файл `base.db` должен оставаться на локальном диске
сервера, а не в общей сетевой папке.

## Как это работает
- Типы форм заданы в `app.py` (`FORM_TYPES`). У каждой записи в таблице `forms`
  есть столбец `form_type` (`030` или `030po`), по которому выбирается шаблон
  и скрипт.
- Маршруты: список — `/forms/<тип>`, создание — `/forms/<тип>/new`,
  открытие/сохранение — `/form/<id>`, печать — `/form/<id>/print`. Тип карты
  определяется по записи в БД.
- Все поля формы при сохранении собираются в один JSON и пишутся в столбец
  `data`. Отдельно дублируются Ф.И.О. и дата рождения — для списка.
- Автосохранение использует маршруты `/forms/<тип>/autosave` и
  `/form/<id>/autosave`. Неполные данные допускаются в черновике; полная
  проверка дат и СНИЛС выполняется при обычном сохранении.
- Столбец `revision` реализует оптимистическую блокировку: обновление проходит
  только при совпадении ревизии в браузере и базе. Конфликт возвращает HTTP 409.
- Динамические блоки диагнозов имеют имена вида `d16_0_diag`, `d17_2_p3_place1`
  и т.п. При открытии карты JavaScript восстанавливает нужное число блоков из
  сохранённых данных и пересчитывает номера пунктов.
- Проверка дат и СНИЛС дублируется на сервере (`validate_form` в `app.py`):
  при некорректных данных карта не сохраняется, а форма возвращается со
  списком ошибок.
